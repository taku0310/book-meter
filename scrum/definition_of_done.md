# 完成の定義（Definition of Done）

> 完成の定義は、インクリメントがプロダクトに必要な品質基準を満たした状態を正式に記述したものである。
> プロダクトバックログアイテムが完成の定義を満たした瞬間にインクリメントが誕生する。
> — スクラムガイド 2020

## 対象プロダクト

**Book Meter**

- **プロダクト名**: Book Meter
- **プロダクト説明**: スタイリッシュで洗練されたUI/UXを通じて、読書体験を豊かにする本管理Webサービス
- **技術スタック**: TypeScript + React + Next.js
- **主な機能**: 本の検索、読了記録、お気に入り登録

## 完成の定義

すべての項目が「はい」と判定できた場合に、そのプロダクトバックログアイテムは「完成」とみなします。

---

### 1. コード品質

| # | 基準 | 判定方法 | 判定 |
|---|------|---------|------|
| 1-1 | TypeScript の型チェックがエラーなく完了している（`tsc --noEmit` が成功している） | 開発環境でコマンド実行確認 | はい / いいえ |
| 1-2 | ESLint ルール（.eslintrc の設定）に準拠し、警告なくコードが実装されている | `npm run lint` コマンド実行確認 | はい / いいえ |
| 1-3 | コードレビューが2名以上の開発者から承認されている | GitHub PR の Review ステータス確認 | はい / いいえ |
| 1-4 | 不要なコメント、デバッグコード、console.log が削除されている | 本番ブランチでのコード目視確認 | はい / いいえ |
| 1-5 | 関数・コンポーネントは単一責任の原則に従い、実装されている | コードレビュー承認時に確認 | はい / いいえ |

### 2. テスト

| # | 基準 | 判定方法 | 判定 |
|---|------|---------|------|
| 2-1 | ユニットテストがすべて成功し、テストカバレッジ 80% 以上である | `npm run test` コマンド実行確認 | はい / いいえ |
| 2-2 | 実装した機能に関連するシナリオテスト（統合テスト）が成功している | QA テスト実施ログで確認 | はい / いいえ |
| 2-3 | 本番ブランチへのマージ後、エンドツーエンドテスト（E2E）が成功している | テスト実行結果レポート確認 | はい / いいえ |
| 2-4 | デグレード（既存機能の不具合）が発生していない | 回帰テスト実施ログで確認 | はい / いいえ |

### 3. ドキュメント

| # | 基準 | 判定方法 | 判定 |
|---|------|---------|------|
| 3-1 | 実装した機能について、README または個別ドキュメントが作成されている | docs フォルダまたは README.md を確認 | はい / いいえ |
| 3-2 | 複雑なビジネスロジックやアルゴリズムについて、設計書またはコメントが記載されている | ドキュメント内容確認 | はい / いいえ |
| 3-3 | API エンドポイント（バックエンド実装時）については、インターフェース定義書が更新されている | API ドキュメント確認 | はい / いいえ |

### 4. 動作確認

| # | 基準 | 判定方法 | 判定 |
|---|------|---------|------|
| 4-1 | 実装した機能が、受入基準をすべて満たしており、正常に動作している | 開発環境での動作確認 | はい / いいえ |
| 4-2 | Chrome、Firefox、Safari の複数ブラウザで動作確認が完了している | ブラウザ互換性テストログ確認 | はい / いいえ |
| 4-3 | 本番環境と同一条件の検証環境でのテストが成功している | ステージング環境での実行結果確認 | はい / いいえ |
| 4-4 | ページロード時間が 100ms 以内である（Core Web Vitals 対応） | Lighthouse または DevTools Performance で測定 | はい / いいえ |

### 5. UI/UX の品質

| # | 基準 | 判定方法 | 判定 |
|---|------|---------|------|
| 5-1 | ミニマリストスタイルのデザイン原則に準拠しており、UI が統一されている | デザインレビュー実施 | はい / いいえ |
| 5-2 | レスポンシブデザイン対応（モバイル・タブレット・デスクトップ）が実装されている | 複数デバイスでの表示確認 | はい / いいえ |
| 5-3 | アクセシビリティ基準（WCAG 2.1 Level A）に準拠している | aXe DevTools などでのスキャン確認 | はい / いいえ |
| 5-4 | ユーザーフィードバック（顧客・佐藤）を反映し、期待に沿っている | 顧客からのサインオフ確認 | はい / いいえ |

### 6. ビルド・デプロイ

| # | 基準 | 判定方法 | 判定 |
|---|------|---------|------|
| 6-1 | ビルドがエラーなく完了している（`npm run build` が成功している） | CI パイプライン実行ログ確認 | はい / いいえ |
| 6-2 | ステージング環境へのデプロイが成功している | デプロイログ確認 | はい / いいえ |
| 6-3 | 本番環境へのデプロイ準備が完了している（デプロイ予定日時・手順が明確） | デプロイプラン確認 | はい / いいえ |

### 7. プロセス・管理

| # | 基準 | 判定方法 | 判定 |
|---|------|---------|------|
| 7-1 | スプリントバックログのタスクステータスが「完了」に更新されている | Backlog またはプロジェクト管理ツールで確認 | はい / いいえ |
| 7-2 | PBI が完了状態に更新され、履歴記録されている | product_backlog.csv で status を Done に更新 | はい / いいえ |
| 7-3 | GitHub PR が主ブランチに マージされている | GitHub リポジトリで PR マージ履歴確認 | はい / いいえ |
| 7-4 | 関連する障害物がすべて解決または記録されている | impediment_log.csv で対応状況を確認 | はい / いいえ |

### 8. セキュリティ

| # | 基準 | 判定方法 | 判定 |
|---|------|---------|------|
| 8-1 | 環境変数に保存するべき秘密情報（API キー、DB パスワード等）がソースコードに含まれていない | コード内容確認、git-secrets スキャン | はい / いいえ |
| 8-2 | 依存パッケージに既知の脆弱性がない（npm audit で HIGH 以上の警告がない） | `npm audit` 実行結果確認 | はい / いいえ |
| 8-3 | SQL インジェクションやクロスサイトスクリプティング（XSS）への対策が施されている | セキュリティレビュー実施確認 | はい / いいえ |

**注記**: MVP（最小実行可能版）段階のため、セキュリティ基準は簡略化しています。本番リリース前に追加のセキュリティ監査が必要です。

## 補足事項

- **MVP 段階の方針**: セキュリティ基準は簡略化していますが、本番リリース前に包括的なセキュリティ監査を実施する予定です。
- **パフォーマンス基準**: ページロード時間 100ms 以内は Google の Core Web Vitals ガイドラインに基づいています。
- **ブラウザ互換性**: Chrome、Firefox、Safari の3種類を標準対応ブラウザとします。
- **検証責任**: コードレビューとテスト実施は開発者（伊藤・田中）の責任です。最終確認はスクラムマスター（高橋）が実施します。

---

## 定義の適用範囲

このDoD はすべての **プロダクト機能実装** に適用されます。

- ✅ 適用対象：新機能実装、バグ修正、UI/UX 改善
- ❌ 非適用：ドキュメント整備のみ、インフラ作業のみ
